## 🎯 戦略：500円で売るには？

> **名称変更予定**: 「hazakura-note」→ **「hazakura editor」**（v0.6 リリースに合わせて変更予定）

現在の状態は **「無料で使うには十分」から「課金するには惜しい」の間**。
500円（約 $3.50）は「衝動買い」の価格帯。ユーザーが「あ、これ500円なら」と思うには、1〜2個の刺さる機能が必要。

### 競合ポジショニング

| 競合 | 価格 | 差別化ポイント |
|------|------|----------------|
| Apple Notes | 無料（エコシステム） | シンプル、iCloud同期 |
| Bear | $2.99/月 | 美しいUI、タグ管理 |
| Typora | $15（買い切り） | WYSIWYG |
| iA Writer | $50（買い切り） | フォーカスライティング |
| Obsidian | 無料 | プラグイン豊富、ナレッジベース |
| **hazakura editor** | **500円（買い切り）** | **美しい和風テーマ、Diff機能、Agent連携** |

**唯一の武器**：
> 「美しい日本語環境の Markdown エディタ × Diff × AI Agent」
> ——この組み合わせは他にない。

### 名称変更の意図

「hazakura-note」から「hazakura editor」へ改名する。

- **狙い**: 古き良き「サクラエディタ」を連想させる方向へ。テキストエディタとしての出自を明確にする
- **タイミング**: v0.6（Foundation Release）に合わせて実施。バージョン名として「名前を変えた回」と覚えてもらいやすくする
- **影響**: README、パッケージ名、tauri.conf.json、リポジトリ名、GitHub Release 名、ドメインなど
- **懸念**: macOS の bundle identifier、brew tap 名、ファイル関連付けの都合。一度変えると二度手間になるので、v0.6 完成時点で `package.json` の name だけ先行変更し、tauri.conf.json の identifier は慎重に判断する

### 「hazakura editor」とサクラエディタの関係

「hazakura editor」は名称の由来通り、**サクラエディタの正統後継を目指すわけではない**。サクラエディタが持つ「軽量・高速・日本語第一」の精神を受け継ぎつつ、Markdown と AI Agent に特化した現代的再解釈を目指す。

#### サクラエディタから学ぶべきもの

| サクラエディタの機能 | hazakura editor での扱い | 時期 |
|---------------------|-------------------------|------|
| **文字コード/改行コードの明示・変換** | ⭐ 採用。ステータスバーに encoding 表示 + 保存時の自動判別。日本語エディタとしての基本。サクラユーザーなら一発で「わかってる」と感じる | v0.7 |
| **名前を付けて保存（別名保存）+ 文字コード選択** | ⭐ 採用。Save As ダイアログに encoding ドロップダウン（UTF-8 / Shift-JIS / EUC-JP）。別名保存時に改行コードも選択可能に。Rust の `encoding_rs` で変換 | v0.7（文字コード表示とセット） |
| **Grep（ファイル横断検索）** | ⭐ 採用。Global Search (Cmd+Shift+F) として v0.7 で予定済み | v0.7 |
| **正規表現置換** | ⭐ 採用。Replace (v0.6) の正規表現モードとして。サクラとの共通言語 | v0.6（Replace の拡張として） |
| **矩形選択** | ✅ CodeMirror 標準搭載。設定有効化のみ | v0.6（Multi-cursor と同時） |
| **キーバインドカスタマイズ** | ⭐ 採用検討。VS Code の keybindings.json 的な仕組み。最もサクラ的な価値 | v0.8 以降 |
| **行ブックマーク** | △ 採用候補。行番号横クリックでブックマーク、F2 で移動。実装軽い | v0.8 以降 |
| **アウトライン解析** | ✅ 済み（OutlinePane）。サクラの「 types 別色分け」までは不要 | 済み |
| **折り返し設定** | ✅ CodeMirror 標準。wordWrap 設定の UI 追加のみ | v0.6（余裕があれば） |
| **対応括弧の強調** | ✅ CodeMirror 標準（bracket matching extension）| 実質0 |
| **diff** | ✅ 済み（DiffPane） | 済み |

#### サクラエディタから意図的に取らないもの

| 機能 | 理由 |
|------|------|
| マクロ（PascalScript / JavaScript） | プラグイン機構に通じる。安全境界を破る。Markdown 特化なら不要 |
| プラグイン機構 | 攻撃面が増える。hazakura editor の「安全」ブランドと相反する |
| アウトラインのタイプ別色分け | テキストエディタとしては強いが、Markdown 特化なら Markdown 文法の構文ハイライトで十分 |
| タグジャンプ / 関数一覧 | C/C++/Java 向け。Markdown エディタでは不要 |
| 文字コード変換ダイアログ | Rust の `encoding_rs` で自動判別＋透過保存するのでダイアログ不要 |

#### サクラエディタユーザーにアピールするには

サクラエディタ（開発停滞気味・Windows専用）からの移行先として hazakura editor を位置づけるなら、以下が必要：

1. **文字コード/改行コードのステータスバー表示** — これがないと「日本語エディタとして半人前」に見られる
2. **キーバインドのカスタマイズ** — サクラユーザーは自分のキーバインドに強いこだわりを持つ
3. **高速起動 / 軽量動作** — 既に Tauri なので Electron よりは速い。ただし初回起動の Rust コンパイル済みバイナリとしては悪くない
4. **Grep の正規表現対応** — サクラの強力な grep は多くのユーザーを惹きつけている

これらは全部、アプリの「安全」思想と矛盾しない範囲で実装可能。

### フェーズ提案：3つの方向性

#### 方向性 A：「安全なAI Markdownエディタ」を深掘り（推奨）

| 機能 | やること | なぜ |

## 外部ファイル名変更への追従 / アプリ内 Rename

### 課題

Finder や mv コマンドでファイル名を変更すると、アプリ内のタブは古いパスを指したままになる。
現在の動作：

```
- タブは path でファイルを識別している
- 外部 rename 後、そのタブを保存しようとすると「ファイルが見つからない」エラーになる
- ツリーは OS のファイル一覧を再取得するので、新しい名前のファイルが別物として表示される
- 結果的に「古いパスのタブ」と「新しい名前のファイル」がアプリ内で分断される
```

### 対応案

| # | 項目 | やること | 工数 |
|---|------|---------|------|
| 1 | **アプリ内 Rename（ツリー右クリック）** | ファイルツリーで右クリック → Rename → インライン編集 → Rust の `fs::rename` で実行。成功時に開いているタブの path を新しい名前に更新する | 1〜2日 |
| 2 | **外部 rename 検出** | タブの path が指すファイルが存在しなくなったとき、ファイルツリーから同名（旧名）のファイルが消えていることを手がかりに「このファイルは移動または名前変更された可能性があります」と表示する。完全な追跡は難しいので、「切り離されたタブ」として表示して手動で再リンク or 閉じる | 2〜3日（検出ロジック要設計） |
| 3 | **Tab path の再解決** | ファイル未検出時に、ファイルツリーの最新状態から「内容が一致するファイル」や「元のファイル名に近いファイル」を候補表示して再リンクできるようにする | 3〜5日（重め） |

**おすすめ**: まず #1（アプリ内 Rename）を実装する。外部 rename への追跡は難易度の割に需要が限られるので、アプリ内で rename できてしまえば回避できる。「rename したいならアプリ内でやる」という運用で OK。

### おまけ：ファイルの移動（ツリー内ドラッグ）

Rename の延長で、ファイルツリー内でのドラッグ＆ドロップによるファイル移動／フォルダ移動もできるとベスト。

```
操作: ツリー内のファイルを別のフォルダに D&D
実装: Rust 側で fs::rename(old_path, new_path) → 成功時にタブの path を更新 + ツリー再取得
```

懸念点:
- Rename + Move で「ファイルツリーが簡易ファイラー化」する。やりすぎると Finder の再実装になりかねない
- 移動先のパスに同名ファイルがあったときの上書き確認が必要
- タブで開いているファイルを移動した場合、タブの path を追従させるロジックが必要

**判断**: Rename までは確実にやる価値がある。Move（D&D移動）は「欲しくなったらやる」でいい。Rename ができれば、ファイル名変更＋必要ならツリー上で階層を合わせる運用でカバーできる範囲。

```
   → ターミナル越しに Agent の出力を読んで入力を送る、という基本体験は既にある

2. RPC で追加される体験が「見た目がきれいになる」程度にとどまる
   → メッセージ・tool log・diff を分類表示しても、それ自体が note アプリのコア価値を変えない
   → むしろ「解釈レイヤー」を持つことで、誤表示・未対応イベントの握りつぶしなどの責任が増える

3. MCP など外部プロトコルの進化は CLI mode でも享受できる
   → pi が MCP ツールを呼べるようになれば、hazakura-note 側は何も変えずに Agent の能力が拡張される
   → RPC を自前実装しなくても、接続先の Agent CLI が賢くなれば済む

4. 工数をかけるなら、エディタ本体の完成度（Cmd+P / 自動保存 / Replace / Global Search）に回したほうが
   note アプリとしての価値が高まる
```

RPC mode は「いつでも作れる選択肢」として残し、現時点では CLI mode の完成度を高める方向に集中する。

### CLI mode の課題と改善計画

CLI mode の操作性には以下の課題がある：

```
a) エディタで読んでいる内容を Agent に渡すのが面倒
   → 選択 → コピー → ターミナルにペースト → 「これについて教えて」と手書き

b) よく使う指示を毎回手打ちしている
   → 「要約して」「英文校正して」「このコードレビューして」

c) Agent の応答が流れて消える
   → あとで見返したいときにスクロールバックしかない

d) Agent が今どのファイルを見ているか知らない
   → 毎回「このファイル読んで」とパスを送る必要がある

e) ポーリングで全出力を毎回 IPC 送信しているため、出力が多いとラグが発生する
   → 現在 200ms ごとに getAll している。差分（last_seen_seq）ポーリングに変更することで大幅改善可能

f) エディタとターミナルのフォーカス移動が不便
   → ショートカットによるペイン切替がない。Tab キーは CodeMirror が消費する
```

これらの課題は、RPC mode に移行しなくても CLI mode のまま改善できる。

#### 改善項目（優先度順）

| # | 項目 | やること | 工数 |
|---|------|---------|------|
| 1 | **差分ポーリング** | Rust 側で `last_seen_seq` パラメータを受け取り、新しい chunk だけ返す。フロントは差分を追記するだけ。シリアライズ負荷と React 再レンダリングが激減 | 半日〜1日 |
| 2 | **選択範囲→Agent 送信** | エディタで選択 → 右クリック「選択範囲を Agent に送信」or ショートカット → 選択テキストがターミナルの stdin に送られる。実装は `editorPaneRef.current.getSelection()` → `onTerminalData()` | 半日 |
| 3 | **プリセットプロンプトボタン** | ターミナル上部に [要約] [校正] [翻訳] [コードレビュー] チップ。クリックでプロンプトテンプレート + 選択範囲を stdin に流す。設定でカスタム追加可能に | 1日 |
| 4 | **ペイン切替ショートカット（Cmd+`）** | エディタ ↔ ターミナルのフォーカスを往復。Escape でエディタに戻る（Zen Mode と同じパターン） | 半日 |
| 5 | **セッションログ保存** | 「会話を保存」ボタン → AgentWorkbenchOutputChunk[] を .md 形式で保存。セッションの内容がファイルとして残る | 半日 |
| 6 | **ファイルコンテキスト自動添付** | メッセージ送信時にアクティブファイルのパスを先頭に付与。「現在のファイル: path/to/file.md」を自動挿入 | 1日 |

合計工数: 3〜4日。RPC mode の設計・実装（2〜3週間）と比較すると大幅に軽い。

#### RPC mode への移行判断条件

CLI mode で上記改善を一通り行ったあと、それでも以下が起きる場合に RPC mode を再検討する：

```
- ターミナルのスクロール性能がどうしても追いつかない
- Agent の出力をアプリ内の DiffPane / PreviewPane などと直接連携させたい需要が強い
- CLI mode の改善だけでは差別化にならないと判断した
```

ただし、その場合も RPC mode は experimental とし、CLI mode を常に fallback として残す構成をとる。
