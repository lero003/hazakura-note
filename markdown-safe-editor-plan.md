Status: Draft

Last reviewed: 2026-05-26

## 1. 概要

本企画は、Markdownを中心に、開発用途の基本的なテキストファイルを安全かつ軽快に開けるシンプルなテキストエディタを開発するための企画案である。

VSCodeのような高機能IDEではなく、拡張機能・LSP・デバッガ・ターミナル統合などを持たない、軽量で安全な「書く・読む・整える・比べる」ための作業環境を目指す。

主な用途は以下の通り。

- Markdown記事や企画書の作成
- 開発用設定ファイルの確認・軽微な編集
- JSON / YAML / HTML / CSS / JS / TS などの基本テキストファイルの閲覧
- AIなどで生成・修正した文章と元ファイルの比較
- 信頼しきれないプロジェクトを、実行せず安全に開く

本エディタの中心思想は、以下である。

> 実行しない。補完しすぎない。勝手に変えない。差分で確認する。

---

## 2. 背景

近年、VSCodeは非常に便利な開発環境である一方、拡張機能を中心に攻撃面が広がっている。

VSCodeの拡張機能は、ワークスペース内のファイル、Git情報、環境変数、ネットワーク、外部コマンドなどにアクセスしうるため、便利さと引き換えにリスクが増している。

また、AntigravityのようなAI開発環境はエージェント機能に強みがある一方、純粋なテキストエディタとしては弱く、単純にテキスト形式のファイルを開く用途では使いづらい。

そのため、以下のようなニーズがある。

- 開発プロジェクト内のファイルを安全に開きたい
- Markdownを快適に書きたい
- 補完やLSPは不要
- 拡張機能に依存したくない
- 任意コード実行を避けたい
- AIが作った修正版と元ファイルを安全に比較したい
- IDEではなく、安心して使える軽いテキスト作業場が欲しい

---

## 3. コンセプト

### 3.1 基本コンセプト

このエディタは、VSCodeの代替ではない。

目的は、IDEを置き換えることではなく、以下のような安全な作業環境を提供することである。

> Markdown中心の、安全なテキスト作業机。

「コードを書くためのIDE」ではなく、「ファイルを安全に読む・書く・整える・比べるためのエディタ」として設計する。

---

### 3.2 目指す価値

- Markdownを書く体験を軽くする
- 開発用ファイルを安全に閲覧・編集できる
- 拡張機能なしで使える
- 任意コード実行をしない
- Gitに依存しない
- AI修正や手作業修正の差分を確認しやすくする
- 小さく、速く、壊れにくいエディタにする

---

## 4. 対象ユーザー

主な対象ユーザーは以下。

- Markdownで記事・企画書・メモを書く人
- VSCodeを使っているが、拡張機能の安全性に不安がある人
- AI生成物やAI修正版を人間の目で確認したい人
- 開発プロジェクト内のファイルを安全に開きたい人
- LSPや補完よりも、軽さと安全性を重視する人
- Gitクライアントではなく、単純なテキスト比較ツールが欲しい人

---

## 5. 非目標

本エディタでは、以下を目指さない。

- VSCode互換
- 拡張機能マーケットプレイス
- LSP対応
- コード補完
- デバッガ
- ターミナル統合
- Gitクライアント機能
- リモート開発
- AIエージェント統合
- プロジェクトビルド
- パッケージ管理
- 任意コマンド実行

この「やらないこと」を明確にすることで、安全性・軽量性・保守性を優先する。

## 6. 基本機能

### 6.1 ファイル操作

- フォルダを開く
- ファイルツリー表示
- テキストファイルを開く

- Cmd+Sで保存

---

### 6.3 エディタ本体

- プレーンテキスト編集
- 行番号表示
- 行折り返し切り替え
- 検索
- 基本的なシンタックスハイライト
- Markdown優先の編集体験

対応候補のファイル形式。

- `.md`
- `.txt`
- `.json`

- `.yml`
- `.html`
- `.css`

- `.tsx`
- `.jsx`
- `.toml`
- `.env`
- `.gitignore`

ただし、コード補完やLSPは行わない。

## 7. Markdown中心機能

### 7.1 Markdownプレビュー

- Markdownプレビュー
- 編集画面との左右分割表示
- 見出しアウトライン
- コードブロック表示
- frontmatter表示
- 画像リンクの簡易確認

HTMLは原則として安全化する。

Markdown内HTMLをそのまま描画するとXSSリスクがあるため、初期状態では無効化またはsanitizeを行う。

---

### 7.2 Markdown記法補助

コード補完は行わないが、Markdownに限定した記法補助は提供する。

対象は以下。

- `**太字**`
- `` `inline code` ``
- `[リンク](url)`
- `![画像](path)`
- 箇条書き
- 番号付きリスト
- チェックボックス
- 見出し
- frontmatter

これは「コードを理解する補完」ではなく、「Markdownの形を崩さないための入力支援」と位置づける。

---

### 7.3 リスト補助

以下のような補助を提供する。

```md
- item
-
```

番号付きリストでは、Enter時に番号を自動で続ける。

```md
1. item
2.
```

空行でリストを終了する。

---

### 7.4 チェックボックス補助

Markdownのタスクリストを扱いやすくする。

```md
- [ ] todo
- [x] done
```

候補機能。

- チェックボックス記法の挿入
- クリックによる `[ ]` / `[x]` の切り替え
- タスク一覧表示

---

### 7.5 Frontmatter補助

記事やメモ用途に向けて、frontmatterテンプレートを挿入できるようにする。

```yaml
---
title:
description:
tags:
created:
updated:
---
```

テンプレートは設定可能にする。

---

### 7.6 内部リンク・画像パス補助

Markdownに限定して、以下の補完を提供する。

- 同一フォルダ内のMarkdownファイルリンク補完
- 画像ファイルパス補完
- 同一ファイル内の見出しアンカー補完

例。

```md
[関連記事](./notes/example.md)
![画像](./images/example.png)
[詳細](#section-name)
```

この補完対象は、Markdownファイルと画像ファイルに限定する。

コード補完には拡張しない。

---

## 8. Markdown lint / 修正支援

### 8.1 基本方針

Markdownの軽い間違い修正はありとする。

ただし、自動修正しすぎると本文の意図を壊す可能性があるため、基本は「提案」または「手動実行」に留める。

方針は以下。

> 提案はする。でも本文の主導権は人間に残す。

---

### 8.2 検出対象

候補となる検出対象。

```md
#見出し
```

を

```md
# 見出し
```

にする提案。

```md
##見出し
```

のスペース不足検出。

```md
[リンク] (url)
```

の不要スペース検出。

```md
-  item
```

の余分なスペース検出。

その他。

- 見出し階層の飛び
- 空行不足
- frontmatterの形式崩れ
- Markdownリンクの閉じ忘れ
- コードフェンスの閉じ忘れ
- 画像altの欠落

---

### 8.3 修正機能名

機能名はAIっぽくしすぎず、控えめにする。

候補。

- Tidy Markdown
- Fix Markdown Syntax
- Format Markdown
- Clean Up
- Proof
- Lint

---

## 9. Diff機能

### 9.1 基本方針

Diff機能は提供する。

ただし、Git連携はしない。

本エディタにおけるDiffは、あくまで「2つのテキストファイルを比較する機能」である。

---

### 9.2 対象外

以下は対象外。

- Git status
- Git diff
- stage / unstage
- commit
- push
- pull
- branch
- merge
- rebase
- conflict解決
- blame
- submodule対応

Gitクライアント化を避ける。

---

### 9.3 ファイル同士の差分

提供する機能。

- 左ファイルを選択
- 右ファイルを選択
- 比較実行
- インラインdiff表示
- サイドバイサイドdiff表示
- 差分箇所へのジャンプ
- 差分数の表示
- 差分結果のコピー
- `.diff` として保存

用途。

- AIが修正したMarkdownと元原稿の比較
- 古い記事と新しい記事の比較
- 設定ファイルの差分確認
- コピーしてきたコード片の差分確認
- 手作業修正前後の比較

---

### 9.4 Markdown向けDiff

Markdown向けには、通常の行単位diffに加えて、将来的に以下のような見せ方を検討する。

- 見出し単位の変更一覧
- frontmatter差分の別表示
- 段落単位diff
- 単語単位ハイライト
- 変更されたセクションへのジャンプ

例。

```txt
変更あり:
- ## はじめに
- ## 料金について
- ## FAQ
```

文章の比較では、コード向けdiffよりも「どの見出し配下が変わったか」が重要になる。

## 10. セキュリティ設計

### 10.1 基本方針

本エディタは、以下の制約を持つ。

- 拡張機能なし
- 任意コード実行なし
- ターミナルなし
- 自動実行なし
- 外部通信なし
- LSPなし
- デバッガなし
- Git操作なし
- スクリプト実行なし
- ワークスペース内のファイルを勝手に解析しすぎない

---

### 10.2 ファイルアクセス

ファイルアクセスはユーザーが選択したフォルダまたはファイルに限定する。

勝手にホームディレクトリ全体を読むような設計にはしない。

---

### 10.3 Markdownプレビュー安全化

Markdownプレビューでは、以下を考慮する。

- HTML無効化またはsanitize
- scriptタグ無効化
- 外部画像読み込みの制限
- 外部リンクを勝手に開かない
- ローカルファイル参照の扱いを明示

---

### 10.4 大容量ファイル

大容量ファイルは、エディタ全体のフリーズやメモリ消費につながる。

初期方針。

- 5MB以上は警告
- 10MB以上は読み取り専用候補
- バイナリ判定されたファイルは開かない
- ログビューア化は目指さない

---

## 11. 技術選定

### 11.1 推奨構成

推奨構成は以下。

```txt
Tauri + CodeMirror 6 + React または Svelte
```

理由。

- Electronより軽量
- ファイルI/Oの権限を絞りやすい
- CodeMirror 6でMarkdown編集・補助・diff表示がしやすい
- UIをWeb技術で素早く作れる
- Rust側を薄く保てる

---

```txt
      diff.ts
      lint.ts
  src-tauri/
    src/

```

## 14. AI支援について

AI支援は将来的には検討可能。

ただし、常時補完や自動書き換えは避ける。

望ましい方針。

```txt
選択範囲を指定
↓
修正候補を生成
↓
diffで確認
↓
人間が適用
```

対象候補。

- 誤字修正
- 表記ゆれ修正
- Markdown記法修正
- 箇条書き整理
- 見出し構造整理
- frontmatter補完

本文の主導権は常にユーザーが持つ。

## 15. 競合・類似ツールとの差別化

### VSCodeとの差別化

VSCodeは高機能だが、拡張機能・LSP・ターミナル・デバッガなどにより攻撃面が広い。

本エディタは、それらを持たない。

差別化ポイント。

- 拡張機能なし
- 任意コード実行なし
- Markdown中心
- Git操作なし
- ファイル同士diffに限定
- 軽量
- 安全性重視

---

### Obsidianとの差別化

ObsidianはMarkdownノート環境として強力だが、ノート管理・リンク管理・プラグイン文化が中心である。

本エディタは、開発用ファイルも安全に開けるテキストエディタ寄りにする。

---

### Typoraとの差別化

Typoraは美しいMarkdownエディタだが、開発プロジェクト内の複数ファイル編集や安全なファイル比較という方向性は弱い。

本エディタは、Markdown編集と開発用テキストファイル閲覧の中間を狙う。

## 17. まとめ

本企画の本質は、VSCodeのような万能IDEを作ることではない。

むしろ、万能性を捨てることで、安全性・軽量性・明確な用途を得る。

このエディタが提供する価値は以下である。

- Markdownを書く
- 基本的な開発用テキストファイルを開く
- Markdown記法を軽く補助する
- 間違いを控えめに直す
- 2つのファイルを比較する
- ただし、コードは実行しない
- Gitも操作しない
- 拡張機能も持たない

最終的なコンセプトは、以下に集約される。

> 安全に開く。静かに書く。差分で確かめる。

小さく始めるなら、まずはMarkdownエディタとして作り、次にMarkdown補助、最後にファイル同士diffを入れるのがよい。

この方向なら、VSCodeの劣化版ではなく、独自の思想を持った安全なテキスト作業環境として成立する。
